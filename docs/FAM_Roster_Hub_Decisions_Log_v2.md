## Design Phase Decisions (March 2026)

These decisions were made during the spec design phase, reviewed by
Claude Opus, and are reproduced here as the baseline.

| Date | Decision | Reason | Impact |
|---|---|---|---|
| March 2026 | Supabase over Firebase | Relational model required for graph-heavy People/Credits/Relationships data | All Phase 3 matching depends on PostgreSQL relational queries |
| March 2026 | Shared deduplicated PEOPLE table across all clients | Enables cross-client Phase 3 matching | Every credit entry must resolve to a canonical People record |
| March 2026 | Controlled vocabularies for all role fields that drive logic | Prevents data rot; CV grouping and sorting requires consistency | CREDIT_ROLES and PROJECT_PEOPLE_ROLES as admin-extendable lookup tables |
| March 2026 | PEOPLE.primary_role is free text (display only) | Not used for grouping or matching; personal descriptive label | Must never be used in queries expecting consistent values |
| March 2026 | PROJECT.sub_type is a lookup table (PROJECT_SUBTYPES) | Prevents "Documentary" vs "Doc" inconsistency appearing on CV output | Admin-extendable; initial values: Short, Documentary, Limited Series, Commercial, Telefilm |
| March 2026 | bio_full stored as sanitised HTML | Enables rich text (bold, italic) in bios and PDF output | Allowed tags: `<p>` `<strong>` `<em>` `<br>` only; Tiptap editor; sanitise on save |
| March 2026 | Project dedup key: title + category, enforced as partial unique index | Year excluded because projects span years or lack confirmed date; database enforcement prevents import script bypassing UI | Import script must handle conflicts explicitly; log duplicates, do not silently create |
| March 2026 | year_start + year_end integer fields replace single year field | Projects span multiple years; some have TBC date at point of entry | CV display logic: single year / range / "In Production" / blank per truth table in spec |
| March 2026 | network_distributor removed from PROJECTS; normalised into PROJECT_COMPANIES | Prevents drift between free text and the Companies table | All networks and distributors stored via PROJECT_COMPANIES with role: Network or Distributor |
| March 2026 | Supabase audit logging for bio version history | Two-slot model (published + draft) sufficient for UI; history preserved at DB level | No separate BIO_VERSIONS table required |
| March 2026 | Soft delete (deleted_at) on all core tables | Hard deletes break referential integrity silently | CLIENTS, PROJECTS, CREDITS, PEOPLE, COMPANIES, AWARDS, PROJECT_PEOPLE, PROJECT_COMPANIES all have deleted_at |
| March 2026 | updated_at on all tables, auto-set via trigger | Required for Optimiser sync and debugging | Every core table has updated_at managed by Supabase trigger |
| March 2026 | ProWee tables (PROWEE_UPLOADS, PROWEE_LISTINGS, PROWEE_MATCHES) defined in Phase 1 migration | Reference Phase 1 entities; adding later requires schema changes to tables with existing data | No UI or parser built until Phase 3; tables exist in schema from day one |
| March 2026 | Awards require admin approval (draft/published); reels do not | Awards are factual claims on professional CVs; reel links are low-risk URLs | Awards follow same gate as credits; reels are client-editable without approval |
| March 2026 | Relationship merge rules: upgrade only, never downgrade; unpublish does not revert | Relationships are historical facts | Explicit rules prevent ambiguous developer decisions during build |
| March 2026 | Unique constraint on RELATIONSHIPS (client_id + person_id) | Merge rules assume at most one record per pair; race conditions otherwise create duplicates | Partial unique index: (client_id, person_id) WHERE deleted_at IS NULL |
| March 2026 | Unique constraint on CREDITS (client_id + project_id) | Prevents silent duplicate creation from import bugs or UI race conditions | Partial unique index: (client_id, project_id) WHERE deleted_at IS NULL |
| March 2026 | Unique constraint on PROJECT_PEOPLE (project_id + person_id + role_on_project) | A person should not be listed in the same role on the same project twice | Partial unique index: (project_id, person_id, role_on_project) WHERE deleted_at IS NULL |
| March 2026 | deleted_at added to PROJECT_PEOPLE and PROJECT_COMPANIES | Required for partial unique index constraints to compile correctly in SQL | Without deleted_at, WHERE deleted_at IS NULL in constraint causes a migration error |
| March 2026 | Deactivated clients: data retained, excluded from Phase 3 and visualisations | Historical data must not be destroyed; inactive clients should not pollute active reports | is_active and deleted_at gate matching and display; underlying data untouched |
| March 2026 | Migration SQL written before UI scaffold | AI scaffold may generate its own schema; database is source of truth | Cursor generates migration from spec; UI conforms to migration |
| March 2026 | Bulk import pipeline built before data-entry UI | ~100 clients cannot be entered manually through a UI | Import schema maps to existing FAM spreadsheet formats; must be confirmed before build session |
| March 2026 | Import schema confirmed before build session begins | Build session should execute an agreed schema, not design one | Minimum: one real client's spreadsheet mapped to import targets and reviewed before session |
| March 2026 | No Lovable; AI Studio or Cursor toolchain | Lovable uses Gemini models accessible free in AI Studio; Cursor preferred per Opus recommendation | If using AI Studio for scaffold: manual Supabase connection required after GitHub export |
| March 2026 | No public profile URL in Hub | Webflow handles public artist pages; Hub is internal and client-facing only | Future: Webflow pulls from Hub via API |
| March 2026 | PitchGen in Phase 2, not Phase 1 | Depends on Phase 1 relationship data being populated before it produces meaningful output | Phase 1 builds the data foundation; PitchGen activates in Phase 2 |
| March 2026 | CV generator is template-based assembly, not AI | Deterministic invoice-style generation; AI adds cost without value for structured output | AI used only for optional pitch bio shortening, triggered explicitly by admin |
| March 2026 | Credits created automatically as by-product of project entry | Users should never interact with CREDITS table directly | Project entry UI creates PROJECT and CREDITS records simultaneously |
| March 2026 | Optimiser API confirmed; integration viable | Verified via Optimiser Touchpoint feature documentation | Next step: obtain API credentials from account manager |
| March 2026 | Decisions log separated from spec into its own file | Prevents spec from growing unwieldy; prevents accidental overwrite of log entries by AI instance updating spec | Log lives as FAM_Roster_Hub_Decisions_Log.md in repo and knowledge base |

[DECISIONS LOG ENTRY]
Date: March 2026
Decision: Added base_locations, nationalities, primary_tax_territory, secondary_tax_territory, manager_name, manager_email to CLIENTS table.
Reason: Onboarding document review revealed these fields are relevant to the Hub for pitch context; financial and legal fields from the same document are excluded as they belong in Optimiser.
Impact: base_locations and nationalities are arrays to support multi-value entries (a client can be based in multiple countries); all country fields use UI-enforced dropdown selection from ISO list, not free text, to maintain consistency.

[DECISIONS LOG ENTRY]
Date: March 2026
Decision: CAE/IPI number and PRO affiliation excluded from CLIENTS table.
Reason: Not required for the Roster Hub's core purposes; may be added in a future release if sync-specific features are developed.
Impact: None on current build scope.

[DECISIONS LOG ENTRY]
Date: March 2026
Decision: "Home country" framing rejected; replaced with base_locations array.
Reason: Composers frequently work across multiple countries and the concept of a single "home" is ambiguous; base_locations accurately reflects where they are based or regularly work.
Impact: UI must support multi-select on this field.

[DECISIONS LOG ENTRY]
Date: April 2026
Decision: Added client_company_relationships table to track direct client-to-company relationships independent of credit history.
Reason: Credit-derived company connections are already queryable via CREDITS → PROJECT_COMPANIES → COMPANIES, but clients may have meaningful relationships with companies (e.g. meetings, warm contacts) that are not reflected in their credit history. These are a real signal in sync representation.
Impact: Purely additive; no changes to existing tables. Phase 3 ProWee company matching deferred: credit-derived company flagging is already supported by existing schema, and a dedicated prowee_company_matches table (if needed) will be designed during Phase 3 when requirements are clearer.
Spec update needed: Yes, Section 4.1 (new table definition) and Section 10 (decisions log table).

Date: April 2026
Decision: Bulk import of credits and projects will use an LLM-assisted conversion pipeline to transform Word-format CVs and credit lists into structured spreadsheets, rather than direct CSV mapping or manual UI entry.
Reason: FAM credit and project data exists as Word documents, not structured spreadsheets. An LLM conversion step is a more efficient and scalable path than manual entry or a bespoke parser.
Impact: The conversion pipeline is a separate build outside the Hub itself, leveraging an LLM (likely Claude) to standardise source documents into the confirmed import schema before upload. The import script and schema mapping 
work defined in Section 4.2 remains valid as the target format; only the source ingestion method changes. Build order is unaffected: import pipeline still precedes data-entry UI for credits and projects.
Spec update needed: Yes, Section 4.2 (note that source documents are Word-format; LLM-assisted conversion is the planned ingestion approach).

Date: April 2026
Decision: Vite adopted as the build tool for the React + TypeScript frontend.
Reason: Vite is the standard build tool for React + TypeScript projects deployed to Vercel; it is fast, well-supported, and requires minimal configuration. No viable alternative was considered.
Impact: All frontend source lives under src/ at the repo root; environment variables exposed to the browser must use the VITE_ prefix; vite.config.ts added to repo root.
Spec update needed: Yes, Section 8 tech stack table (add Vite row).

Date: April 2026
Decision: Tailwind CSS and shadcn/ui adopted as the UI styling and component framework.
Reason: Tailwind provides utility-first CSS without writing custom stylesheets; shadcn/ui provides pre-built, accessible React components (tables, forms, dialogs, dropdowns) that accelerate admin UI development significantly. Both are well-supported with React + Vite + TypeScript.
Impact: All styling uses Tailwind utility classes; reusable UI components sourced from shadcn/ui rather than built from scratch; tailwind.config.ts, components.json, and src/components/ui/ added to repo. Design tokens (colours, spacing) can be customised via Tailwind config as the FAM brand template is finalised.
Spec update needed: Yes, Section 8 tech stack table (add Tailwind and shadcn/ui rows).
